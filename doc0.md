Feasibility Assessment
Short answer: Very feasible, and it's the right call architecturally. Here's the full breakdown by component.

1. Nuxt 4 + Nuxt UI 3 (Monitoring App)
Feasibility: High. Since you're rebuilding from scratch there's no migration burden — you just start fresh on the latest versions. Nuxt UI 3 is a significant rewrite of v2 (built on Reka UI primitives) but for a new project that's irrelevant.

2. App Split — Monitoring vs Reporting
Feasibility: High. This is the right architecture. The two concerns have fundamentally different requirements:

Monitoring (Nuxt 4)	Reporting (Laravel + Filament)
Users	Field workers, evaluators	Programme managers, admins
Connectivity	Offline-first, mobile	Always online
DB	PouchDB + CouchDB	MySQL
Data flow	Write-heavy	Read-heavy, complex queries
Deployment	Capacitor (iOS/Android) + PWA	Web only
3. Laravel + Inertia + Filament
Feasibility: High. Filament 3 is an excellent fit:

Built-in tables with server-side filtering, sorting, export
Stats/chart widgets (ApexCharts under the hood) for dashboards
Role-based access out of the box
MySQL means you can write proper aggregate queries for district/facility/tool breakdowns — something CouchDB is terrible at
Inertia gives you a Vue 3 frontend if you need custom interactive views beyond what Filament provides.

4. CouchDB → MySQL Sync
Feasibility: High. This is the most novel piece but CouchDB makes it well-suited via its _changes feed.

Recommended approach — Laravel Artisan sync command:


CouchDB /_changes?feed=normal&since=<last_seq>&include_docs=true
    → Laravel queue worker / scheduled command (every 1-5 min)
    → Transform JSON → MySQL rows
    → Store last_seq in sync_checkpoints table (resumable)
This lives entirely inside the Laravel codebase, no extra service to deploy. For reporting, near-real-time (1-5 min lag) is perfectly acceptable. The _changes feed with a persisted seq number makes it fully resumable after downtime.

Schema design is the real work — flattening CouchDB documents into relational tables:


evaluations → evaluation_sessions → evaluation_item_scores
users ← evaluations → mentees
districts → facilities → users
5. No Max Sessions
Feasibility: High — and it actually simplifies things.

The current fixed session_1...session_5 keys in the data model are replaced with an ordered array. In CouchDB: sessions: [{ number: 1, date: ..., scores: [] }, ...]. In MySQL: a evaluation_sessions table with no row limit. This is cleaner.

6. Updated Evaluation Item Data (from the doc)
This is where there are some significant structural surprises beyond just wording changes:

a) Dual scoring — Every item now has TWO scores per session, not one:

Mentee score: 1–5 (same scale as before, + N/A/0)
Mentor score: 1 (I had to do it) → 5 (I did not need to be there)
This changes the data model, the evaluation UI, and the MySQL schema.

b) Advanced competency flag — Grey items are "not required for basic competency." Items need an is_advanced flag.

Key Design Decisions to Make Early
Dual scoring UI — Do the evaluator score and mentee score get entered simultaneously in one form, or in two separate steps? This affects the evaluation wizard flow significantly.

Counselling section (DC1-DC9) — Is this scored once per session regardless of tool, or once per tool? The doc is ambiguous.

Auth between apps — The simplest split: Filament has its own admin auth for the reporting app, with data read from MySQL (synced from CouchDB). No shared session. Evaluators only interact with the Nuxt app.

Sync latency — Is a 1-5 minute CouchDB→MySQL lag acceptable for the reporting platform? Almost certainly yes for this use case.
